## 1. 概述

### 1.1 项目背景

AgentEngine-dev 项目已有完整的 Redis 缓存模块实现（`src/cache/` 目录），包含 Redis 客户端、缓存管理器、序列化器、监控指标等核心组件。然而，当前缓存集成存在以下问题：

1. **缓存一致性风险**：本地缓存和 Redis 缓存更新顺序不一致，多实例部署时存在数据不一致问题
2. **降级机制不完善**：Redis 故障时虽有降级逻辑，但缺少熔断机制，可能导致雪崩效应
3. **监控告警缺失**：虽有 Prometheus 指标，但缺少告警规则和可视化大盘
4. **性能优化空间**：缓存预热、批量操作、本地缓存淘汰策略等未实现

### 1.2 项目目标

**总体目标**：在现有 Redis 缓存模块基础上，进行增量增强，提升缓存一致性、可用性和可观测性，最终实现高性能、高可用的分布式缓存体系。

**具体目标**：
1. 解决多实例部署下的缓存一致性问题
2. 实现完善的熔断和降级机制，防止缓存故障引发系统雪崩
3. 建立完整的监控告警体系，实现故障快速发现和定位
4. 优化缓存性能，降低数据库压力

### 1.3 项目范围

**包含范围**：
- ✅ 增强现有缓存模块（`src/cache/`）
- ✅ 优化配置管理器的缓存策略（`src/core/config_manager.py`）
- ✅ 新增熔断器组件
- ✅ 新增监控告警规则
- ✅ 新增缓存预热功能

**不包含范围**：
- ❌ 数据库层优化（单独项目）
- ❌ Agent 业务逻辑优化（单独项目）
- ❌ 前端监控大盘开发（单独项目）

## 2. 业务需求

### 2.1 核心业务需求

#### [REQ_REDIS_CACHE_001] 缓存一致性保障（增强/修复）

**需求类型**：存量模块增强  
**优先级**：P0（最高）  
**来源**：SA 存量解剖报告 §4.1  

**需求描述**：
解决多实例部署时本地缓存与 Redis 缓存不一致的问题。

**验收标准**：
1. ✅ 配置更新时，所有实例的本地缓存在 5 秒内失效
2. ✅ 多实例部署环境下，配置读取一致性达到 99.9%
3. ✅ 配置更新延迟不超过 1 秒
4. ✅ 提供手动清除所有实例缓存的管理接口

**技术方案**：
- 使用 Redis Pub/Sub 实现缓存失效广播
- 保留本地缓存作为 L1，但实现 LRU 淘汰策略
- 配置更新时先更新数据库，再删除 Redis，最后广播失效消息

**影响范围**：
- `src/cache/cache_manager.py`（新增 Pub/Sub 订阅）
- `src/core/config_manager.py`（更新缓存失效逻辑）
- `src/api/router.py`（新增缓存清除接口）

- **数据一致性**：配置更新后 5 秒内所有实例生效

- **监控指标**：提供至少 10 个核心监控指标
- **告警延迟**：异常发生到告警触发不超过 1 分钟

## 3. 技术方案

### 3.1 缓存架构设计

#### 3.1.1 多级缓存架构（增强）

```
┌─────────────────────────────────────────────────────────────────┐
│                        应用层                                    │
│  (AgentManager, ConfigManager, etc.)                           │
└───────────────────────────┬─────────────────────────────────────┘
                            │
                ┌───────────┴──────────┐
                │                      │
                ▼                      ▼
        ┌──────────────┐       ┌──────────────┐
        │   L1 缓存     │       │   L2 缓存     │
        │  本地内存     │       │    Redis     │
        │  (TTLCache)  │       │  (分布式)     │
        │              │       │              │
        │ 容量: 1000   │       │  容量: 无限   │
        │ TTL: 300s    │       │  TTL: 3600s  │
        └──────┬───────┘       └──────┬───────┘
               │ miss                  │ miss
               └──────────┬────────────┘
                          │
                          ▼
                  ┌──────────────┐
                  │   L3 数据库   │
                  │  (PostgreSQL)│
                  └──────────────┘
```

**缓存策略**：
- **Read-Through**：优先读 L1 → L2 → L3，逐级回填
- **Write-Through**：写数据库 → 删 L2 → 删 L1（广播）
- **Cache-Aside**：更新时删除缓存，而非更新缓存

#### 3.1.2 缓存一致性方案（新增）

**方案一：Redis Pub/Sub 广播（推荐）**

```
配置更新流程：
1. 更新数据库
2. 删除 Redis 缓存
3. 发布缓存失效消息到 Redis Channel
4. 所有订阅的实例收到消息后删除本地缓存
```

**方案二：短暂 TTL + 最终一致性**

```
配置更新流程：
1. 更新数据库
2. 删除 Redis 缓存
3. 本地缓存设置较短 TTL（30s）
4. 最多 30s 后自动生效
```

**选择**：采用方案一，适用于对一致性要求较高的场景。

#### 3.1.3 熔断器设计（新增）

```
熔断器状态机：
          错误率 > 50%
    ┌─────────────────────┐
    │                     │
    ▼                     │
┌────────┐  30s后    ┌────────┐
│ CLOSED │ ─────────>│  OPEN  │
└────────┘           └────┬───┘
    ▲                     │
    │     试探请求成功     │
    └───────── ┌──────────┘
              ▼
         ┌──────────┐
         │HALF_OPEN │
         └──────────┘
              │
              │ 试探请求失败
              └─────────> OPEN
```

**关键参数**：
- **错误阈值**：50%（可配置）
- **时间窗口**：10 秒（滑动窗口）
- **恢复时间**：30 秒
- **试探请求数**：3 个

### 9.2 术语表

| 术语 | 说明 |
|------|------|
| L1 缓存 | 本地内存缓存，容量小、速度快 |
| L2 缓存 | Redis 分布式缓存，容量大、共享 |
| Read-Through | 读取时自动回填缓存的策略 |
| Write-Through | 写入时同时更新缓存的策略 |
| Cache-Aside | 更新时删除缓存的策略 |
| 熔断器 | 防止级联故障的保护机制 |
| LRU | 最近最少使用淘汰算法 |
| TTL | Time To Live，缓存过期时间 |
| Pub/Sub | 发布订阅模式 |
